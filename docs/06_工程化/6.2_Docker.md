# 1. 为什么在 AI 工程中使用 Docker

容器把 **Python 版本、系统库、CUDA 运行时、业务依赖** 打成可复现单元，在开发机、测试机与 **Kubernetes** 上行为一致，避免「本地能跑、线上缺库」。对 **LLM 推理、RAG 网关、Embedding 服务** 等，Docker 还便于 **按进程隔离 GPU 占用** 与 **滚动升级**。

# 2. GPU 与基础镜像

## 2.1. NVIDIA Container Toolkit

宿主机需安装与驱动匹配的 **NVIDIA Container Toolkit**，运行容器时加 **`--gpus all`**（或 Compose / K8s 中等价配置）。镜像内的 **CUDA 小版本** 不宜高于宿主机驱动支持的上限，否则会出现运行时错误。

## 2.2. 镜像选型建议

- **推理**：优先选用 **PyTorch / vLLM / TGI** 等官方或社区维护的 **CUDA 标签**，与框架文档中的矩阵一致，避免自拼 `ubuntu + 任意 cuda + pip torch` 导致 ABI 问题。  
- **仅 CPU**：用 slim 系基础镜像即可，缩小体积与攻击面。  
- **开发调试**：可用带完整工具链的镜像；**生产** 尽量用精简 runtime 镜像。

# 3. Dockerfile 实践

## 3.1. 多阶段构建

- **构建阶段**：安装编译依赖、执行 `pip install -r requirements.txt` 或 `poetry export` 后安装。  
- **运行阶段**：只拷贝 **虚拟环境** 或 `site-packages`、应用代码与入口脚本，减少层数与敏感工具（如 gcc）残留。

## 3.2. 依赖层与代码层分离

把 **`requirements.txt` / `pyproject.toml` 的 COPY 与 pip install** 放在单独早期层，**业务代码 COPY** 放后面。这样改代码时仍能命中缓存，避免每次全量重装依赖。

## 3.3. `.dockerignore`

排除 `.git`、`__pycache__`、`.venv`、测试缓存、大体积 **模型权重与数据**（改由卷挂载或启动时下载）。否则构建上下文过大、镜像层膨胀且易误把密钥打进上下文。

# 4. 模型权重与数据卷

- **不要**把数十 GB 权重写入镜像层：构建慢、分发难、回滚成本高。  
- 使用 **bind mount / 命名卷**、**init 容器** 或 **对象存储挂载**，运行时只读挂载可降低误删风险。  
- 多副本只读共享同一卷或同一对象存储前缀时，注意 **文件锁与一致性**（部分推理框架会写缓存）。

# 5. 运行与编排要点

## 5.1. `docker run` 常见参数

- **端口映射**：`-p 8000:8000`。  
- **环境变量**：`-e CUDA_VISIBLE_DEVICES=0`、API Key 等（生产优先用 **secret 机制**，勿写进镜像）。  
- **资源限制**：`--memory`、`--cpus` 防止旁路服务拖垮宿主机。  
- **健康检查**：在镜像内提供 **`/health`** 等端点，Compose / K8s 用 **healthcheck / readiness** 与业务语义对齐。

## 5.2. Docker Compose（本地/单机）

适合 **API + 向量库 + Redis** 等多容器联调：统一网络 DNS、卷声明与环境文件 **`.env`**。AI 场景常为 **一个 GPU 服务容器 + 若干 CPU 依赖容器**；注意 **GPU 只能被需要算力的容器声明**，避免多个推理容器抢同一卡。

# 6. 安全与可观测性

- **非 root 用户**运行应用进程（`USER` 指令 + 正确文件权限）。  
- 镜像内 **不硬编码** 长期密钥；使用运行时注入。  
- 日志输出到 **stdout/stderr**，由编排层采集；避免在容器可写层堆积巨型日志。

# 7. 踩坑记录

- **无 GPU 的机器上构建 GPU 镜像**：部分 `docker build` 会在构建期执行需要 CUDA 的步骤 → 可拆成 **CPU 构建 + 仅运行时 GPU 镜像**，或在与线上一致的构建机上构建。  
- **层缓存被一条 `COPY . .` 破坏**：先拷依赖清单装包，再拷源码。  
- **时区与编码**：统一 `TZ`、`LANG`，避免日志与文件名乱码。  
- **SSE / WebSocket**：经反向代理时需注意 **缓冲与超时**（与 [6.1 FastAPI](6.1_FastAPI服务化.md) 中的网关配置呼应）。
